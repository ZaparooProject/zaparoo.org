# AGENTS.md

Instructions for AI coding agents working on this repository.

## Project overview

Zaparoo.org is the documentation website for the Zaparoo project, built with Docusaurus. It covers the core software, the companion app, hardware drivers, and community resources.

## Stack

- **Docusaurus** 3.9.2 (TypeScript config)
- **React** 19
- **TypeScript** ~6.0.2
- **Node** ≥18
- **Package manager:** pnpm

## Commands

```bash
pnpm start          # dev server, binds to 0.0.0.0
pnpm build          # static build into build/
pnpm typecheck      # tsc — run this to check types
pnpm serve          # serve the built site locally
pnpm clear          # clear Docusaurus cache
```

Do not run `pnpm build` speculatively — only run it when explicitly asked or when verifying a specific issue.

## Repository layout

```
docs/           Versioned documentation (autogenerated sidebar)
blog/           Date-prefixed blog posts (YYYY-MM-DD-slug/index.mdx)
src/pages/      Static custom pages (contact, downloads, showcase, etc.)
src/components/ Reusable React components
static/         Images and other static assets
versioned_docs/ Older doc versions
```

## Remote content

`docs/core/dev/` and `docs/core/api/` are populated at build time by `docusaurus-plugin-remote-content`, which pulls markdown from the zaparoo-core GitHub repo. Both directories are gitignored (except their `_category_.json` files). Never edit files inside them by hand — changes will be overwritten.

## Writing style

These rules apply to all user-facing content: docs pages, blog posts, PR descriptions, and release notes.

- Never add "See Also" sections
- Never use emojis unless explicitly asked
- Never use subjective words in section titles (Recommended, Best, Preferred, etc.) — keep titles neutral and descriptive
- Keep tone casual and conversational, matching the existing blog post style
- Focus on what users need to know; skip exhaustive technical detail
- Defer to other docs pages for detailed configuration rather than duplicating it
- Never guess implementation details — only document what can be verified from source code

## Linking

- Link liberally, Wikipedia-style — any concept, feature, or term with a dedicated page is a candidate
- First mention only — link once per page, not on every occurrence
- Prefer internal links; check whether a docs page exists before linking externally
- External links are fine for GitHub repos, specs, and external tools

## Writing voice

Apply these rules when writing any user-facing prose. The goal is direct, human-sounding text — not AI-shaped content.

- No inflated-significance words: *testament, pivotal, enduring, vital, crucial, tapestry, landscape* (abstract use)
- No promotional filler: *vibrant, rich, seamless, nestled, breathtaking, must-visit, groundbreaking*
- No superficial -ing tails: *…ensuring…, …highlighting…, …reflecting…, …showcasing…*
- No vague attributions: *"industry reports", "experts believe", "some observers"*
- No em dashes where a comma or full stop works fine
- No rule of three when one or two ideas suffice
- No negative parallelism: *"not just X, it's Y"*
- No boilerplate challenge/outlook sections
- No sycophantic openers: *"Great question!", "Certainly!", "Of course!"*
- Vary sentence length and structure; short punchy sentences are fine
- Prefer `is`/`has` over `serves as`/`boasts`/`stands as`
- Use first person when it fits; opinions and reactions are allowed
- Be specific: concrete details beat vague claims

## Boundaries

**Always:**
- Use pnpm for all package management commands
- Follow the writing style and linking rules above for all user-facing content
- Leave `docs/core/dev/` and `docs/core/api/` untouched — they're rebuilt from zaparoo-core

**Ask first:**
- Adding new dependencies
- Changing the structure of `docusaurus.config.ts`
- Adding or modifying redirects

**Never:**
- Run `pnpm build` unless explicitly asked
- Edit files in `docs/core/dev/` or `docs/core/api/`
- Claim a feature exists without verifying it in the relevant source code
- Commit secrets or API keys
